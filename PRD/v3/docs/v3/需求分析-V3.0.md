# LANChat 产品需求分析（V3.0）

> 文档状态：V3.0 产品化需求基线  
> 当前代码基线：V3.0.0 开发版 / `master`
> 更新日期：2026-07-19

## 1. 文档目的

本文定义 LANChat V3.0 的产品目标、用户角色、使用场景、功能需求、非功能需求、边界与验收标准。V3.0 建立在 V2.3 已实现通信、文件、协作与多实例能力之上，重点解决“如何安装、如何发现、如何更新、如何管理、如何验证”的产品化问题。

本文不把规划项视为已实现。实际完成情况以《实施状态-V3.0.md》及代码、测试、构建产物为准。

## 2. 产品定位

LANChat 是面向校园、工厂、办公室、项目现场和应急环境的 **LAN-first 私有即时协作系统**：

- 优先在局域网和组织自有基础设施中运行；
- 外网中断时，局域网内的核心协作能力仍可继续；
- 消息、文件、账号、日志与运行数据由部署方控制；
- 支持 Web、桌面、Android 和 iOS 多端接入；
- 网络恢复后可完成消息补拉、任务恢复和受控同步。

## 3. 背景与问题

V2.3 已经解决核心通信链路，但产品使用仍存在以下阻力：

1. 浏览器首次访问需要已知节点地址，不能直接监听 mDNS；
2. 用户需要手动打开浏览器，缺少托盘、系统通知、单实例和开机自启；
3. 管理员依赖命令行、Compose 和分散页面完成部署与诊断；
4. 当前没有稳定的安装包、签名、自动更新和正式 Release 流程；
5. Android/iOS 尚未形成可分发客户端；
6. 自动化测试、指标与 Trace 尚未覆盖完整产品链路；
7. 离线文件任务、草稿与应用重启恢复仍需增强。

## 4. 产品目标与非目标

### 4.1 V3.0 产品目标

| 编号 | 目标 | 成功定义 |
|---|---|---|
| OBJ-01 | 可安装 | 普通用户无需开发环境即可安装桌面或 Android 客户端 |
| OBJ-02 | 可发现 | 客户端可自动发现局域网节点，并保留二维码、历史地址和手动输入回退路径 |
| OBJ-03 | 可持续运行 | 桌面端支持托盘、通知、单实例、重连与可选开机自启 |
| OBJ-04 | 可更新 | 桌面端可安全检查、下载、校验并安装新版本 |
| OBJ-05 | 可管理 | 管理员可查看节点、依赖、日志、版本、存储和诊断信息 |
| OBJ-06 | 可验证 | 关键通信、断网恢复、多实例和安装构建具备自动化回归 |
| OBJ-07 | 可观测 | 关键接口、WebSocket、依赖和任务具有健康、指标、日志与 Trace 证据 |
| OBJ-08 | 多端复用 | Web、Tauri、Capacitor 复用同一套核心 Vue UI 和协议模型 |

### 4.2 非目标

V3.0 不承诺：

- 独立数据库节点双向复制或多主同步；
- 自动冲突合并和跨组织联合身份；
- 端到端加密；
- 完全冷启动离线登录；
- 本地 AI 助手；
- 微服务拆分；
- 未通过压测验证的 99.9% SLA 或固定并发指标。

## 5. 用户角色

| 角色 | 主要目标 | 权限边界 |
|---|---|---|
| 普通用户 | 发现节点、登录、聊天、收发文件、使用临时房间和广播 | 仅访问自身与所属会话数据 |
| 群主/房间管理员 | 管理群组或临时房间成员与策略 | 仅管理授权范围内的会话 |
| 广播发布者 | 向授权接收者发布应急信息 | 普通发布者只能选择有效好友；系统管理员权限更高 |
| 系统管理员 | 创建账号、查看诊断、管理节点、日志、版本和部署 | 敏感操作必须再次鉴权和审计 |
| 部署管理员 | 安装服务端、配置依赖、备份、升级和恢复 | 可接触密钥、数据库和对象存储，必须遵守最小权限 |
| 外部访客 | 通过受限房间码参与临时协作 | V3.0 可选扩展，不默认开放完整账号能力 |

## 6. 核心使用场景

### SCN-01 校园临时协作

老师或学生在校园局域网启动节点，其他设备自动发现并加入。外网不可用时，局域网聊天、文件与房间继续运行。

### SCN-02 工厂与项目现场

现场电脑运行 LANChat Server，员工通过桌面或 Android 客户端加入。管理员通过管理 GUI 查看节点健康、存储容量和运行日志。

### SCN-03 办公室私有部署

组织通过 Docker Compose 部署双应用实例、MySQL、Redis、MinIO 和 Nginx，客户端通过统一地址访问，并由自动更新机制维护桌面版本。

### SCN-04 应急网络

临时局域网内快速启动节点，客户端通过 mDNS 或二维码连接。授权人员发布应急广播，接收者查看并确认。

### SCN-05 网络抖动与恢复

客户端断网后将文本消息写入本地发件箱，网络恢复后自动重连、补拉遗漏消息并继续未完成上传任务。

## 8. 非功能需求

### 8.1 性能

- 不在无压测证据时写死并发承诺。
- 局域网文本消息应以用户不可感知延迟为目标，测试报告记录 P50/P95/P99。
- 桌面端空闲状态需控制 CPU、内存与网络心跳开销。
- 大文件采用流式哈希、固定分片和受控并发，避免一次性读入内存。
- 管理日志、指标和诊断接口必须设定返回条数与扫描字节上限。

### 8.2 可靠性

- `CHAT_ACK` 仅在数据库事务提交后返回。
- Redis Pub/Sub 失败不能破坏 MySQL 中的可靠消息事实。
- 更新失败后应继续运行旧版本。
- 应用重启、网络切换和节点短暂不可达后可恢复连接与本地任务。
- 对象删除失败进入持久化重试，不以日志替代删除意图。

### 8.3 安全与隐私

- 桌面原生权限遵循最小授权和显式 scope。
- Sidecar 只能执行白名单命令，不接受任意 shell 字符串。
- 更新包必须进行签名校验。
- 发布密钥不得进入仓库、构建日志或普通 artifact。
- Android cleartext 仅作为受控内网兼容方案，正式部署优先 HTTPS/WSS。
- iOS 需要解释本地网络用途，权限拒绝后提供手动地址与设置引导。
- 诊断包必须脱敏 Token、Cookie、密码、密钥和签名 URL。

### 8.4 兼容性

| 平台 | V3.0 目标 |
|---|---|
| macOS | 本轮 P0 开发与验收主平台，最低系统版本、签名、公证和本地网络权限可追踪 |
| Windows | 纳入构建与 Release 矩阵，覆盖受支持的 WebView2 环境 |
| Linux | 提供至少一种通用发行产物，明确系统 WebView 依赖 |
| Android | Android 优先内测，具体最低版本在工程初始化时锁定并记录 |
| iOS | Beta，需具备 Xcode、签名与本地网络权限条件 |
| Web | 保留 V2.3 同源托管与现代浏览器访问 |

### 8.5 可维护性

- 保持模块化单体服务端，不为 V3.0 强制拆微服务。
- Web、Desktop、Mobile 共享类型、API 客户端和核心状态模型。
- 原生能力通过适配器接口隔离，Web 环境提供降级实现。
- 配置必须区分 dev、test、release，敏感值使用环境变量或受保护 secret。
- 文档、代码和实施状态必须同步更新。

## 10. 验收总标准

### 10.1 本轮桌面 P0 代码验收

本轮“代码已实现”与“正式发布完成”分开判定：

1. Tauri 壳可编译，单实例、关闭隐藏、托盘、通知、自启和白名单深链均有代码与自动测试证据；
2. 未知节点首次连接必须经过协议、节点 ID、`/api/v1/node/info` 和健康端点校验；
3. mDNS 不可用时，缓存节点和手动 HTTP(S) 地址仍可回退，二维码属于后续 P1；
4. 桌面 Refresh Cookie 由 Rust 原生客户端保存，JavaScript 只接触 Access Token；
5. 节点切换同时更新 REST、WebSocket、资源地址与本地缓存属主；
6. 通用 CI、三平台桌面 smoke、双实例/断网 E2E 和草稿 Release 均存在可执行定义；
7. 没有真实 Secrets/证书、签名产物、公证记录和升级回归时，REL-003/REL-004 只能记录为“进行中”。

### 10.2 V3.0 Release Candidate 验收

V3.0 发布候选版本必须同时满足：

1. **可安装**：至少 Windows 与 Android 有可重复构建产物。
2. **可连接**：桌面端可通过 mDNS 或回退路径连接节点。
3. **可协作**：V2.3 的登录、聊天、文件、房间和广播主链路在壳中通过回归。
4. **可恢复**：断网文本、消息补拉和上传恢复通过自动测试。
5. **可更新**：桌面端完成签名更新验证与失败回退测试。
6. **可管理**：管理员可以查看节点、依赖、日志与诊断信息。
7. **可观察**：健康和至少一组核心指标可被采集。
8. **可追踪**：版本、构建、提交和 Release 产物之间可以对应。
9. **状态真实**：README 与实施状态不把未完成项标记为已实现。

## 11. 风险与约束

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 企业 Wi-Fi 禁止多播 | mDNS 发现失败 | 手动地址、二维码、历史节点、管理员预配置 |
| 局域网使用 HTTP | 移动系统限制与窃听风险 | 优先 HTTPS；Android 仅受控例外；界面明确安全状态 |
| Tauri 系统 WebView 差异 | 跨平台显示或 API 行为差异 | 平台构建矩阵、兼容测试、锁定最低系统要求 |
| 签名证书成本与保管 | 无法正式分发或泄露密钥 | 受保护发布环境、最小权限、密钥轮换和离线备份 |
| 多端同时开发 | 人力分散 | macOS 桌面 P0 先形成发布证据，再推进 Android 与 iOS Beta |
| 深离线安全模型复杂 | 凭证泄露或状态误导 | V3.0 只承诺热离线增强，冷启动本地解锁受控试验 |
| GitHub runner 无法测试多播 | 自动化覆盖不足 | 自托管 runner、局域网实验机、夜间测试 |
