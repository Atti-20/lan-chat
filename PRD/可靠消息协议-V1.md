# LanChat WebSocket 可靠消息协议 V1

## 1. 连接与认证

客户端连接：

```text
ws://<node>/ws/chat
```

连接成功后 10 秒内发送 `AUTH`。令牌不得放入查询参数。

```json
{
  "version": 1,
  "event": "AUTH",
  "requestId": "req_01",
  "timestamp": 1784077200000,
  "payload": {
    "token": "<access-token>"
  }
}
```

认证成功返回 `AUTH_OK`；令牌过期返回 `TOKEN_EXPIRED`；设备会话被吊销返回 `FORCE_LOGOUT`。

## 2. 统一信封

```json
{
  "version": 1,
  "event": "CHAT_SEND",
  "requestId": "req_02",
  "clientMsgId": "client_01",
  "conversationId": "private:7:9",
  "timestamp": 1784077200000,
  "payload": {}
}
```

| 字段 | 规则 |
|---|---|
| `version` | 当前固定为 `1` |
| `event` | 大写事件名 |
| `requestId` | 请求型事件必填，用于关联响应和日志 |
| `clientMsgId` | `CHAT_SEND` 必填；同一发送者范围内唯一 |
| `conversationId` | 私聊或群聊事件必填 |
| `timestamp` | 客户端毫秒时间，仅用于诊断，不参与消息顺序 |
| `payload` | 事件数据对象 |

## 3. 发送与确认

私聊发送：

```json
{
  "version": 1,
  "event": "CHAT_SEND",
  "requestId": "req_02",
  "clientMsgId": "client_01",
  "conversationId": "private:7:9",
  "timestamp": 1784077200000,
  "payload": {
    "toUserId": 9,
    "contentType": "text",
    "content": "hello",
    "isBurn": false
  }
}
```

群聊把 `toUserId` 替换为 `groupId`，会话 ID 使用 `group:<id>`。

事务提交成功后返回：

```json
{
  "version": 1,
  "event": "CHAT_ACK",
  "requestId": "req_02",
  "clientMsgId": "client_01",
  "conversationId": "private:7:9",
  "timestamp": 1784077200108,
  "payload": {
    "clientMsgId": "client_01",
    "messageId": "server_01",
    "conversationId": "private:7:9",
    "sequence": 128,
    "serverTime": 1784077200108,
    "duplicated": false
  }
}
```

ACK 超时可以重发，但必须复用相同 `clientMsgId`。服务端若已保存，会返回原 `messageId` 和 `sequence`，同时令 `duplicated=true`。

接收端通过 `CHAT_DELIVER` 获得完整消息。服务端时间和 `sequence` 是最终排序依据。

## 4. 重连同步

认证成功后，客户端为每个会话提交本地最后接收的序列：

```json
{
  "version": 1,
  "event": "SYNC_REQUEST",
  "requestId": "req_sync_01",
  "timestamp": 1784077300000,
  "payload": {
    "positions": {
      "private:7:9": 127,
      "group:3": 42
    },
    "limit": 100
  }
}
```

响应包含缺失消息、服务端最新位置、拒绝访问的会话和分页标记：

```json
{
  "version": 1,
  "event": "SYNC_RESPONSE",
  "requestId": "req_sync_01",
  "timestamp": 1784077300100,
  "payload": {
    "messages": [],
    "latestPositions": {
      "private:7:9": 130,
      "group:3": 42
    },
    "deniedConversationIds": [],
    "hasMore": false
  }
}
```

当 `hasMore=true` 时，客户端保存本批最大序列并使用新位置继续请求。

## 5. 其他事件

| 事件 | 方向 | 用途 |
|---|---|---|
| `PING` / `PONG` | 双向 | 心跳和 RTT |
| `CHAT_READ` | 双向 | 推进会话最后已读序列 |
| `CHAT_RECALL` | 双向 | 两分钟内撤回自己的消息 |
| `CHAT_BURN` | 双向 | 焚毁有权访问的阅后即焚消息 |
| `TYPING_START` / `TYPING_STOP` | 双向 | 短期输入状态 |
| `ONLINE_LIST` | 服务端到客户端 | 已脱敏在线成员列表 |
| `PRESENCE_CHANGED` | 服务端到客户端 | 上下线状态变化 |
| `FRIEND_CHANGED` | 服务端到客户端 | 好友关系变化 |
| `PROFILE_UPDATED` | 服务端到客户端 | 用户昵称或头像变化 |
| `ROOM_STATUS_CHANGED` | 服务端到客户端 | 临时房间成员、状态或策略变化 |
| `BROADCAST` | 服务端到客户端 | 新广播及当前用户回执状态 |
| `BROADCAST_ACK` | 客户端到服务端 | 广播送达确认 |
| `BROADCAST_UPDATED` | 服务端到客户端 | 广播查看/确认状态或撤销状态变化 |
| `BROADCAST_PERMISSION_UPDATED` | 服务端到客户端 | 当前账号的广播发布权限变化 |
| `FILE_TRANSFER_OFFER` | 客户端到服务端、服务端到客户端 | WebRTC 直传任务和 SDP Offer |
| `FILE_TRANSFER_READY` | 服务端到客户端 | 传输任务创建成功及最终任务 ID |
| `FILE_TRANSFER_ANSWER` | 客户端到服务端、服务端到客户端 | 接收设备认领任务并返回 SDP Answer |
| `FILE_TRANSFER_STARTED` | 客户端到服务端 | DataChannel 已开始传输 |
| `FILE_TRANSFER_COMPLETE` | 客户端到服务端、服务端到客户端 | 直传完整性报告及完成通知 |
| `FILE_TRANSFER_FAILED` / `FILE_TRANSFER_FALLBACK` | 客户端到服务端 | 直传失败并申请节点中转 |
| `FILE_TRANSFER_RELAY_COMPLETE` | 客户端到服务端 | 节点中转上传完成 |
| `FILE_TRANSFER_RELAY_COMPLETED` | 服务端到客户端 | 节点中转任务已完成 |
| `FILE_TRANSFER_REJECTED` / `FILE_TRANSFER_CANCELED` | 服务端到客户端 | 传输任务被拒绝、失效或由其他设备认领 |
| `AUTH_OK` | 服务端到客户端 | 连接认证完成，返回脱敏用户/设备信息 |
| `TOKEN_EXPIRED` / `FORCE_LOGOUT` | 服务端到客户端 | Token 过期或设备会话被吊销 |
| `ERROR` | 服务端到客户端 | 结构化业务或协议错误 |

## 6. 错误处理

`ERROR.payload` 包含：

```json
{
  "code": "BUSINESS_REJECTED",
  "message": "无权在该会话发送消息",
  "retryable": false
}
```

若错误携带 `clientMsgId`，客户端应更新对应本地消息，而不是新增系统消息。`retryable=false` 的权限或参数错误应保留为失败状态，等待用户修改或取消。

## 7. 当前限制

- V2.3 支持同一逻辑节点内、共享 MySQL/Redis/MinIO 的多个 Spring Boot 实例；不支持各自拥有独立数据库的节点之间双向复制和冲突合并；
- `CHAT_DELIVER` 已提供，精细送达回执尚未单独落表；
- 单次同步最多处理 100 个会话、全局最多返回 200 条（每会话上限同为 200 条）；
- 文件可通过单请求接口或 V2.3 分片上传会话完成，再通过可靠消息引用；完全离线的文件发件箱仍未实现。

## 8. 多实例路由语义

同一逻辑节点的每个应用实例只直接持有接入本机的 WebSocket Session。目标用户连接在其他实例时，事件通过 Redis Pub/Sub 发布到集群频道，由目标实例写入本机 Session。路由事件包含唯一事件 ID、来源实例、目标类型/目标 ID、业务事件和时间信息；消费端去重，来源实例不会重复消费自己的事件。

跨实例路由不会改变客户端 V1 信封，也不要求客户端知道实际接入实例。下列保障需要区分：

| 事件类别 | 保障方式 |
|---|---|
| `CHAT_SEND` / `CHAT_ACK` / `CHAT_DELIVER` | MySQL 事务、`clientMsgId` 幂等和会话 `sequence` 是最终事实；Pub/Sub 负责低延迟投递，遗漏后由 `SYNC_REQUEST` 补齐 |
| 撤回、焚毁、已读、广播等持久业务状态 | 数据库状态为准；实时通知丢失后重新查询或重连恢复 |
| 输入状态、Presence 等瞬时状态 | Redis 快路径；允许在 Redis 短暂失败时降级，不写入消息历史 |
| WebRTC Offer/Answer/完成通知 | 通过目标设备所在实例实时路由；参与设备离线或路由超时后按文件传输规则降级 |

全局 Presence 使用 Redis 中带 TTL 的记录和周期续约。记录只包含用户、实例、Session 和设备标识，不保存 Access Token、Refresh Token 或完整用户正文。实例异常退出后，未续约记录会自然过期。

Redis 暂时不可用时，本实例已经建立的连接和 MySQL 持久化消息仍可工作；跨实例瞬时事件可能延迟或丢失。Redis 恢复后实例重新订阅，可靠消息由客户端重连同步补偿。Pub/Sub 本身不应被描述成持久消息队列或独立节点数据同步。

## 9. V2.3 分片上传协议

文件字节不放入 WebSocket 帧，而是使用鉴权 REST 接口：

```text
POST   /api/v1/file/uploads
GET    /api/v1/file/uploads/{uploadId}
PUT    /api/v1/file/uploads/{uploadId}/parts/{partNumber}?sha256=<partSha256>
POST   /api/v1/file/uploads/{uploadId}/complete
DELETE /api/v1/file/uploads/{uploadId}
```

创建请求：

```json
{
  "clientUploadId": "upload_client_01",
  "conversationId": "private:7:9",
  "fileName": "report.pdf",
  "fileSize": 7340032,
  "fileType": "application/pdf",
  "fileHash": "<完整文件SHA-256>"
}
```

创建、查询和上传分片响应中的会话数据：

```json
{
  "uploadId": "upload_server_01",
  "status": "UPLOADING",
  "chunkSize": 5242880,
  "totalParts": 2,
  "uploadedParts": [1],
  "expiresAt": "2026-07-17T12:00:00",
  "completedFile": null
}
```

`clientUploadId` 在同一用户范围内用于安全重试创建请求。分片编号采用 `1..totalParts`，分片请求体为 `application/octet-stream` 原始字节；`sha256` 是该分片哈希，不是完整文件哈希。相同编号、大小和哈希的重复提交是幂等成功，冲突内容不能覆盖已有分片。

`complete` 只有在全部分片存在、合并大小和完整 SHA-256 匹配，并且扩展名、MIME、文件头/容器结构复核通过后才直接返回 `FileUpload`；已完成会话被再次创建或查询时，会在上面的 `completedFile` 字段返回同一结果。取消接口不返回业务数据。上传会话归属和原会话权限在查询、上传、完成、取消时都会重新校验。完成后的存储名才可以进入 `CHAT_SEND` 附件内容。

分片、最终文件和缩略图的物理删除不属于 WebSocket V1 信封。业务事务会先把删除意图写入 `file_object_cleanup_task`，事务提交后执行幂等删除；LOCAL/MinIO 暂时失败时保留任务并退避重试。该清理队列只保障对象生命周期，不是聊天事件 Outbox，也不改变 Redis Pub/Sub 与 MySQL 消息同步的职责边界。
