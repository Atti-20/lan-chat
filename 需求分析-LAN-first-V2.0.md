## 1. 文档概述

### 1.1 编写目的

本文档用于明确 LanChat 的产品定位、目标用户、使用场景、功能边界、业务规则、非功能要求和版本规划，作为后续功能分析、系统设计、数据库设计、接口设计、测试设计和项目排期的统一依据。

本文档描述的是“需要实现什么”和“达到什么效果”，不代表所有功能均已完成。各功能的实际完成状态应在 README 或项目进度文档中单独标注。

### 1.1.1 V2.3.0 实施快照

截至 V2.3.0，代码已实现本文档中的可靠消息、文本离线发件箱、文件安全、WebRTC 直传与中转降级、临时房间、应急广播，以及以下新增工程能力：

- 上传会话、分片 SHA-256、相同分片幂等、已上传分片查询、断点续传、合并后的完整大小/SHA-256/内容复核；
- LOCAL 与 MinIO 私有文件对象存储切换，历史 LOCAL 对象按元数据中的实际存储类型继续读取；
- 共享 MySQL、Redis、MinIO 的单逻辑节点多 Spring Boot 实例，使用 Redis Pub/Sub 跨实例路由实时事件，并使用带 TTL 的 Redis Presence 聚合全局在线状态；
- Redis 短暂不可用时保留本实例连接和 MySQL 持久化，可靠消息在重连后通过会话序列和 `SYNC_REQUEST` 补偿。

这里的“多实例”不是本文“可选节点同步”所描述的独立节点数据复制。V2.3.0 尚未实现两套独立数据库之间的双向同步、同步任务、权限传播、冲突检测和冲突合并。实际证据和部署方式以 [README.md](README.md) 与 [实施状态-LAN-first-V2.0.md](实施状态-LAN-first-V2.0.md) 为准。

### 1.2 产品定位

LanChat 是一套面向校园、工厂、办公室、项目现场和应急场景的 **LAN-first 私有即时协作系统**。

系统以局域网通信为优先路径，在互联网不可用、网络不稳定或组织不允许业务数据上传公有云时，仍可提供消息沟通、群组协作、文件传输、通知确认和本地数据留存能力；当设备或局域网节点恢复互联网连接后，可根据组织策略选择不同步、手动同步或自动同步。

### 1.3 产品愿景

让小型组织和临时团队在“无公网、弱网络、数据不可外传”的环境中，仍然能够快速建立可控、可靠、可追踪的协作网络。

### 1.4 核心价值

1. **局域网优先**：同一局域网内优先完成消息和文件传输，降低对公网和第三方云服务的依赖。
2. **断网可用**：客户端断网后可继续查看本地记录、编辑消息并加入待发送队列，恢复连接后自动或手动补发。
3. **数据可控**：支持部署在组织自有电脑、服务器、NAS 或内网环境中，数据由组织自行保存和管理。
4. **可靠通信**：通过消息唯一标识、服务端确认、顺序控制、幂等去重和断线补拉，降低消息丢失、重复和乱序风险。
5. **可选同步**：组织可根据安全要求选择本地独立运行、人工同步或跨节点自动同步。
6. **快速协作**：支持通过组织邀请码、房间码、二维码或局域网节点地址快速加入协作空间。
7. **可诊断**：向用户和管理员展示连接状态、当前通信路径、同步状态及异常原因。

### 1.5 产品边界

LanChat V2.0 的核心目标不是替代微信、QQ、钉钉或企业微信，也不以开放社交、内容社区、支付、直播和复杂办公审批为主要方向。

项目优先解决以下问题：

- 私有环境中的即时通信；
- 网络中断后的消息恢复；
- 局域网中的文件交换；
- 多设备和多成员状态同步；
- 临时协作空间管理；
- 数据生命周期和访问权限控制；
- 可部署、可监控、可测试的工程实现。

以下能力不属于首期强制范围：

- 公网手机号短信注册；
- 开放式陌生人社交；
- 音视频会议；
- 复杂工作流审批；
- 原生 iOS、Android 客户端；
- 完整端到端加密；
- 跨地域多活容灾；
- 百万级用户规模。

## 2. 术语定义

| 术语 | 定义 |
|---|---|
| LAN-first | 系统在同一局域网可用时优先使用本地节点或局域网传输路径，不依赖公网完成核心协作 |
| LanChat 节点 | 部署 LanChat 服务端的计算设备，可以是个人电脑、服务器、工控机或 NAS |
| 组织空间 | 由一个组织管理的一组用户、设备、会话、文件和安全策略 |
| 本地独立模式 | 节点不连接任何外部同步服务，数据只保存在当前局域网 |
| 混合同步模式 | 局域网内优先通信，恢复公网后按策略将指定数据同步到其他节点或中心服务 |
| 客户端离线 | 客户端无法连接当前 LanChat 节点，但仍可使用本地缓存和离线发件箱 |
| 节点离线 | LanChat 节点无法访问公网，但局域网内客户端仍可连接节点并通信 |
| 离线发件箱 | 客户端暂存未成功提交至服务端的消息和文件任务 |
| 服务端确认 ACK | 服务端完成消息校验和持久化后，向发送客户端返回的成功确认 |
| 会话序列号 | 服务端为每个会话内的消息分配的递增顺序号，用于排序和遗漏补拉 |
| 临时协作房间 | 有明确有效期、成员范围和数据清理规则的临时会话空间 |
| 应急广播 | 由授权人员向指定范围发送的高优先级通知，可要求接收者确认 |
| 多应用实例 | 同一逻辑节点中，共享 MySQL、Redis 和 MinIO、但分别持有本机 WebSocket Session 的 Spring Boot 进程 |
| 可选节点同步 | 独立 LanChat 节点之间的数据复制与冲突处理；属于目标能力，V2.3.0 尚未实现 |

## 3. 目标场景与用户

### 3.1 目标场景

#### 3.1.1 校园场景

- 实验室、社团、宿舍或临时活动组在校园网内建立协作空间；
- 教室或机房互联网不稳定时进行资料分发；
- 教师向班级广播通知并统计确认情况；
- 校园活动现场通过二维码快速加入临时房间。

#### 3.1.2 工厂场景

- 生产区域不允许业务数据上传公有云；
- 车间、仓库和办公室通过厂区局域网传递生产通知和文件；
- 班组长发送高优先级通知并查看成员确认状态；
- 网络中断时继续记录消息，连接恢复后统一补发。

#### 3.1.3 办公室场景

- 小型团队在自有服务器或 NAS 上部署内部通信工具；
- 内部文件只允许在组织网络内访问；
- 管理员控制成员、设备、数据保存周期和下载权限；
- 互联网故障时仍保持局域网通信。

#### 3.1.4 项目现场场景

- 建筑工地、展会、赛事、拍摄现场等临时网络环境；
- 成员通过临时房间、二维码或房间码快速加入；
- 现场照片、图纸和任务通知在局域网内快速交换；
- 项目结束后按策略归档或销毁房间数据。

#### 3.1.5 应急场景

- 公网不可用但现场局域网仍可建立；
- 指挥人员向全部或部分成员发送应急广播；
- 接收者确认“已收到”“已执行”或“需要支援”；
- 管理员查看在线状态、未确认人员和最后通信时间。

### 3.2 目标用户

| 用户类型 | 主要诉求 |
|---|---|
| 普通成员 | 快速收发消息和文件，断网后不丢草稿和待发内容 |
| 班组长/项目负责人 | 创建群组或临时房间，发布通知，查看成员确认状态 |
| 组织管理员 | 管理组织成员、角色、设备、数据策略和安全配置 |
| 节点管理员 | 部署和维护 LanChat 节点，查看运行状态、存储和网络诊断信息 |
| 临时访客 | 使用邀请码、房间码或二维码加入指定临时房间 |
| 审计人员 | 在授权范围内查看操作记录、广播确认和文件访问日志 |

## 4. 产品运行模式

### 4.1 本地独立模式

适用于内网、涉密或不允许数据外传的环境。

要求：

- 核心功能不依赖公网；
- 用户、消息、文件和日志全部保存在本地节点；
- 不调用外部短信、邮件、推送或在线预览服务；
- 管理员可导出备份，但系统不得自动向公网发送业务数据；
- 节点公网断开时，局域网内通信不受影响。

### 4.2 局域网优先模式

适用于办公室、校园和工厂等长期局域网环境。

要求：

- 客户端优先连接本地 LanChat 节点；
- 消息通过本地节点完成持久化和分发；
- 文件优先使用局域网路径上传、下载或设备间直传；
- 本地路径不可用时，根据组织策略降级至服务器中转；
- 客户端应明确展示当前通信路径。

### 4.3 混合同步模式

适用于多个办公地点、项目现场回传或需要远程备份的环境。

组织管理员可以选择：

| 同步策略 | 行为 |
|---|---|
| 禁止同步 | 数据永不离开当前节点 |
| 手动同步 | 仅管理员主动执行时同步 |
| 定时同步 | 在配置的时间窗口同步指定数据 |
| 自动同步 | 网络恢复后自动同步允许的数据 |
| 仅元数据同步 | 只同步用户、会话索引和状态，不同步消息正文及文件 |
| 指定范围同步 | 只同步指定组织、房间、消息类型或文件类型 |

同步过程必须提供：

- 同步范围提示；
- 同步进度；
- 成功、失败和冲突统计；
- 可重试任务；
- 审计记录；
- 敏感数据排除策略。

### 4.4 客户端离线模式

当客户端无法连接 LanChat 节点时：

- 用户可查看已缓存的会话和消息；
- 用户可继续编辑文本消息和草稿；
- 发送操作进入离线发件箱；
- 待发送消息显示明确状态；
- 文件任务可暂停并保留进度；
- 恢复连接后根据用户和组织策略自动或手动补发；
- 重复补发不得产生重复消息。

## 5. 用户角色与权限

| 角色 | 核心权限 |
|---|---|
| 组织所有者 | 创建组织、转让组织、配置最高级别安全和同步策略 |
| 组织管理员 | 管理成员、角色、设备、组织设置、数据保留和审计范围 |
| 节点管理员 | 管理节点配置、存储、备份、网络、监控和版本升级 |
| 房间所有者 | 管理指定群组或临时房间、成员、公告、权限和生命周期 |
| 房间管理员 | 协助管理成员、禁言、公告和广播 |
| 普通成员 | 在授权会话中收发消息、文件和确认通知 |
| 访客 | 在指定有效期和指定房间内使用有限功能 |
| 只读成员 | 查看消息和文件，不允许发送或修改内容 |
| 审计员 | 查看授权范围内的审计和访问记录，不参与普通聊天 |

### 5.1 权限原则

- 默认拒绝，按角色和资源授权；
- 前端展示权限仅用于交互，最终权限必须由服务端校验；
- 组织权限、房间权限和文件权限分别判断；
- 房间管理员不得修改组织级安全和同步策略；
- 访客权限必须有有效期和范围限制；
- 敏感操作应记录操作者、时间、设备和结果。

## 6. 核心业务流程

### 6.1 创建并加入私有组织

1. 节点管理员部署并初始化 LanChat 节点；
2. 系统生成组织初始化入口；
3. 组织所有者设置组织名称、安全策略和同步模式；
4. 管理员创建用户、导入成员或生成邀请码；
5. 用户通过节点地址、二维码或客户端发现功能连接节点；
6. 用户完成身份验证并进入组织空间。

### 6.2 局域网发送消息

1. 客户端生成全局唯一的 `clientMsgId`；
2. 消息在本地显示为“发送中”；
3. 客户端通过当前连接将消息发送至 LanChat 节点；
4. 服务端校验身份、会话成员关系和发送权限；
5. 服务端持久化消息并分配会话序列号；
6. 服务端返回 ACK；
7. 发送端将消息状态更新为“已发送”；
8. 服务端向接收设备推送消息；
9. 接收端确认送达或已读；
10. 服务端同步更新相关设备状态。

### 6.3 断网发送与恢复

1. 客户端检测到连接中断；
2. 用户发送的消息进入离线发件箱；
3. 消息显示“等待连接”，允许取消或编辑；
4. 网络恢复后客户端重新认证并请求同步；
5. 服务端根据会话序列号返回遗漏消息；
6. 客户端先完成遗漏消息合并和去重；
7. 客户端按创建顺序补发待发送消息；
8. 服务端依据 `clientMsgId` 保证幂等；
9. 用户可查看补发结果和失败原因。

### 6.4 局域网文件传输

1. 用户选择文件；
2. 客户端计算文件哈希并检查文件是否已经存在；
3. 服务端校验用户是否有权在目标会话发送文件；
4. 系统优先选择局域网传输路径；
5. 支持直传时建立设备间传输通道；
6. 无法直传时降级为节点中转或对象存储；
7. 大文件采用分片传输并保存进度；
8. 完成后校验文件完整性；
9. 服务端创建文件消息和访问授权；
10. 接收者在授权范围内预览或下载。

### 6.5 临时协作房间

1. 用户创建临时房间；
2. 设置房间有效期、成员上限、加入方式和数据清理策略；
3. 系统生成房间码或二维码；
4. 成员加入并获得限定权限；
5. 到期前系统提醒房间所有者；
6. 到期后根据策略冻结、归档或销毁数据；
7. 所有生命周期操作形成审计记录。

### 6.6 应急广播与确认

1. 授权人员选择组织、群组或成员范围；
2. 编辑广播内容，设置优先级和确认要求；
3. 系统向在线设备实时推送；
4. 离线成员上线后优先展示未确认广播；
5. 成员选择“已收到”“已执行”或“需要支援”；
6. 发送者查看确认比例、未确认人员和最后响应时间；
7. 广播状态变化实时同步至授权设备。

## 7. 功能需求

## 7.1 组织与节点管理

### 7.1.1 组织初始化

- 首次部署后必须完成组织初始化；
- 支持设置组织名称、标识、默认语言和时区；
- 支持配置组织数据保留期限；
- 支持选择本地独立、局域网优先或混合同步模式；
- 初始化完成后关闭公开初始化入口；
- 组织所有者可转让所有权，但不得直接删除最后一个所有者账号。

### 7.1.2 节点连接

MVP 阶段支持：

- 手动输入节点地址；
- 扫描节点二维码；
- 使用组织提供的短连接码；
- 保存常用节点；
- 显示节点名称、地址、证书状态和最近连接时间。

后续版本可支持：

- 桌面客户端通过 mDNS/局域网广播发现节点；
- 多节点列表；
- 节点优先级和自动切换；
- 节点健康检查。

V2.1 已实现 Java 服务端的 mDNS 广播/发现和 Web 端服务端结果展示；这里的“桌面客户端发现”仍特指浏览器无法直接完成的 UDP 5353 扫描与零种子首次定位。

### 7.1.3 节点状态

节点管理员可以查看：

- 节点在线时间；
- 当前连接用户数和设备数；
- WebSocket 连接数；
- 消息处理量；
- 待处理同步任务；
- 文件存储占用；
- 数据库和 Redis 状态；
- 当前网络模式；
- 最近异常和告警；
- 应用版本和升级状态。

## 7.2 用户、身份与设备

### 7.2.1 用户创建

私有部署默认采用以下方式：

- 管理员创建账号；
- 批量导入账号；
- 组织邀请码注册；
- 临时访客码加入。

手机号、短信验证码和第三方 OAuth 登录作为可选扩展，不作为本地独立模式的强制依赖。

### 7.2.2 登录认证

- 支持账号密码登录；
- 支持短期 Access Token 与可撤销 Refresh Token；
- 支持登录失败次数限制；
- 支持设备标识；

- 网络恢复时客户端可使用有效会话恢复连接。

### 7.2.3 设备管理

设备列表至少展示：

- 设备名称；
- 设备类型；
- 操作系统和客户端版本；
- 最近登录时间；
- 最近活动时间；
- 最近连接节点；
- 当前在线状态；
- 当前设备标识；
- 是否为可信设备。

用户可：

- 修改设备名称；
- 下线指定设备；
- 撤销可信状态；
- 查看新设备登录记录。

### 7.2.4 访客账号

- 必须绑定指定组织或房间；
- 必须设置有效期；
- 可限制消息、文件、下载和邀请权限；
- 到期后自动失效；
- 可由管理员提前撤销；
- 访客不得访问其他未授权空间。

## 7.5 可靠消息系统

### 7.5.1 消息唯一性

- 每条客户端消息必须包含唯一 `clientMsgId`；
- 服务端以用户和 `clientMsgId` 组合进行幂等校验；
- 客户端重试不得产生重复消息；
- 服务端生成独立消息 ID；
- 同一会话内分配递增序列号。

### 7.5.2 消息状态

发送方可见状态：

- 草稿；
- 等待连接；
- 发送中；
- 已发送；
- 已送达；
- 已读；
- 发送失败；
- 已撤回；
- 已焚毁；
- 同步冲突。

### 7.5.3 消息确认

- 服务端只有在完成必要校验和持久化后才返回 ACK；
- 未收到 ACK 的消息可重试；
- 服务端应返回服务端消息 ID、会话序列号和服务端时间；
- ACK 丢失时，客户端重发同一 `clientMsgId`，服务端返回原消息结果。

### 7.5.4 消息顺序

- 会话内以服务端序列号作为最终顺序依据；
- 客户端本地创建时间只用于临时展示；
- 离线补发后，服务端可以重新确定最终位置；
- 客户端应平滑修正消息顺序，避免重复闪动。

### 7.5.5 遗漏消息补拉

- 客户端保存每个会话最后确认的序列号；
- 重连后按序列号请求遗漏消息；
- 服务端支持分页返回；
- 客户端完成去重、排序和状态合并；
- 同步失败时允许重试，不得清空本地未发送数据。

### 7.5.6 消息撤回

- 默认允许发送者在 2 分钟内撤回；
- 撤回时限由组织管理员配置；
- 撤回操作必须校验发送者身份、会话权限和时间窗口；
- 撤回后保留占位记录；
- 文件消息撤回后撤销对应访问授权；
- 物理文件是否删除由引用计数和数据保留策略决定。

### 7.5.7 阅后即焚

- 支持文本、图片和短视频；
- 文件和语音默认不支持；
- 支持查看后倒计时销毁；
- 支持离开会话后立即销毁；
- 焚毁状态在多设备间同步；
- 本地独立模式下由本地节点执行销毁；
- 截屏提醒只作为具备系统能力的客户端扩展，Web 端不得宣称能够阻止截屏。

## 7.6 断网可用与本地数据

### 7.6.1 本地缓存

客户端可缓存：

- 最近会话；
- 最近消息；
- 草稿；
- 离线发件箱；
- 上传任务；
- 最后同步序列号；
- 用户基本信息；
- 节点连接信息。

缓存数量和保存期限由用户或组织策略控制。

### 7.6.2 离线发件箱

- 消息进入离线发件箱后必须显示明确状态；
- 用户可取消、编辑或重试；
- 补发顺序默认按本地创建时间；
- 对已失效会话、已被移出群组或权限变化的消息，服务端应拒绝并返回明确原因；
- 失败消息保留在本地，禁止无提示丢弃。

### 7.6.3 冲突处理

可能的冲突包括：

- 消息在一个设备撤回，另一个设备仍处于离线状态；
- 用户离线期间被移出群组；
- 文件权限在上传过程中被撤销；
- 多设备同时修改草稿或会话设置；
- 同一数据在两个节点分别修改。

冲突处理原则：

1. 权限和安全状态以服务端最新状态为准；
2. 消息最终顺序以会话序列号为准；
3. 已撤回、已焚毁和已删除状态优先于本地缓存；
4. 用户编辑内容发生冲突时，尽量保留本地副本；
5. 无法自动解决时必须提示用户，不得静默覆盖。

## 7.7 文件传输与文件协作

### 7.7.1 文件类型

支持配置白名单，默认包括：

- 图片：jpg、jpeg、png、gif、webp；
- 文档：doc、docx、xls、xlsx、ppt、pptx、pdf、txt；
- 压缩包：zip、rar、7z；
- 其他类型由管理员配置。

### 7.7.2 文件大小

- 默认单文件上限 100 MB；
- 上限由节点管理员配置；
- 大于阈值的文件自动使用分片上传；
- 局域网直传和节点中转可设置不同上限。

### 7.7.3 秒传

- 客户端计算 SHA-256；
- 服务端检查物理文件是否存在；
- 即使物理文件存在，也必须重新校验当前用户和目标会话的引用权限；
- 秒传只减少重复上传，不自动授予文件访问权。

### 7.7.4 分片与断点续传

- 创建上传任务；
- 查询已上传分片；
- 并发上传缺失分片；
- 暂停和恢复；
- 合并分片；
- 校验完整文件哈希；
- 失败重试；
- 超时任务清理。

### 7.7.5 传输路径

系统可选择：

1. 局域网设备直传；
2. 本地节点中转；
3. 本地对象存储；
4. 组织允许的远程节点或中心存储。

路径选择应考虑：

- 发送者和接收者是否在线；
- 是否处于同一局域网；
- 文件大小；
- 安全策略；
- 客户端能力；
- 传输失败次数。

### 7.7.6 文件访问控制

- 文件与消息、房间和组织权限关联；
- 下载和预览前必须鉴权；
- 支持短期签名地址；
- 支持访问次数和有效期限制；
- 支持禁止访客下载；
- 支持撤销文件权限；
- 支持文件访问日志；
- 原始文件不得放置在可绕过鉴权的静态目录中。

### 7.7.7 在线预览

首期支持：

- 图片预览；
- PDF 浏览器内预览；
- 文本轻量预览。

后续支持：

- Office 文档通过本地 OnlyOffice 或其他私有预览服务转换；
- 视频预览；
- 代码高亮；
- 预览失败自动降级为下载。

本地独立模式不得强制调用公网预览服务。

## 8. 非功能性需求

## 8.1 可靠性

- 已收到服务端 ACK 的消息不得因普通进程重启而丢失；
- 客户端重复发送同一 `clientMsgId` 不得生成重复消息；
- 重连后能够按照会话序列号补拉遗漏消息；
- 离线发件箱数据不得在未提示用户的情况下丢失；
- 文件完成传输后必须校验完整性；
- 同步任务必须可重试；
- 权限撤销应优先于普通消息同步。

## 8.2 性能目标

以下为阶段性验收目标，不代表当前已实现结果。

### MVP 单实例基准目标

- 同时在线 WebSocket 连接数：500；
- 持续消息处理能力：100 条/秒；
- 局域网文本消息 ACK 延迟 P95：不高于 300 ms；
- 普通历史消息分页查询 P95：不高于 500 ms；
- 断线后 5 秒内开始重连；
- 1000 条遗漏消息在正常局域网环境下 10 秒内完成同步；
- 文件上传速度尽量接近实际局域网吞吐上限，不设置虚假的固定速度指标。

### 后续优化目标

- 单节点在线连接数达到 2000；
- 持续消息处理能力达到 500 条/秒；
- 支持同一逻辑节点的多实例 WebSocket 路由（V2.3.0 已具备功能，容量与故障指标仍需实际压测）；
- 以压力测试报告中的实际数据为准。

## 8.3 可用性

- 节点公网断开时，局域网核心功能继续可用；
- Redis、文件预览等非核心服务异常时提供明确降级；
- 预览不可用时允许下载；
- 桌面通知不可用时不影响消息接收；
- 文件直传失败时可以降级为节点中转；
- 服务重启后能够恢复持久化消息和未完成任务；
- 管理员可执行备份和恢复。

## 8.4 安全

- 生产环境使用 HTTPS/WSS；
- 密码使用 BCrypt 或同等级密码哈希；
- Access Token 短期有效；
- Refresh Token 可轮换和撤销；
- WebSocket 连接必须完成身份认证；
- 不在访问日志中记录长期有效的敏感 Token；
- 校验 WebSocket Origin；
- 防止 SQL 注入、XSS、CSRF、越权访问和路径穿越；

## 10. 接口与通信原则

### 10.1 REST API

- 统一前缀 `/api/v1`；
- 统一响应结构；
- 使用明确错误码；
- 历史消息使用游标或序列号分页；
- 所有资源访问必须校验组织和资源权限；
- 文件下载和预览使用授权接口；
- 管理接口与普通用户接口分离。

示例响应：

```json
{
  "code": 200,
  "data": {},
  "msg": "success",
  "requestId": "req_xxx"
}
```

### 10.2 WebSocket

统一事件信封至少包含：

```json
{
  "version": 1,
  "event": "CHAT_SEND",
  "requestId": "req_xxx",
  "clientMsgId": "client_msg_xxx",
  "conversationId": "conv_xxx",
  "timestamp": 1784077200000,
  "payload": {}
}
```

核心事件包括：

- AUTH；
- AUTH_OK；
- PING；
- PONG；
- CHAT_SEND；
- CHAT_ACK；
- CHAT_DELIVER；
- CHAT_READ；
- CHAT_RECALL；
- CHAT_BURN；
- TYPING_START；
- TYPING_STOP；
- SYNC_REQUEST；
- SYNC_RESPONSE；
- BROADCAST；
- BROADCAST_ACK；
- FORCE_LOGOUT；
- TOKEN_EXPIRED；
- ERROR。

### 10.3 文件接口

至少包括：

- 文件存在性检查：`POST /api/v1/file/check`；
- 创建或幂等恢复上传会话：`POST /api/v1/file/uploads`；
- 查询已上传分片：`GET /api/v1/file/uploads/{uploadId}`；
- 上传原始二进制分片：`PUT /api/v1/file/uploads/{uploadId}/parts/{partNumber}?sha256=...`；
- 合并并完整复核：`POST /api/v1/file/uploads/{uploadId}/complete`；
- 取消上传：`DELETE /api/v1/file/uploads/{uploadId}`；
- 文件/分片对象删除失败时保留持久化清理任务并自动重试；
- 获取下载授权；
- 获取预览授权；
- 查询传输状态。

## 11. 异常与降级场景

| 场景 | 预期处理 |
|---|---|
| 客户端与节点断开 | 进入离线状态，消息进入发件箱，自动重连 |
| 节点无法访问公网 | 局域网内继续工作，暂停外部同步 |
| 重复发送消息 | 服务端基于 `clientMsgId` 返回原结果 |
| ACK 丢失 | 客户端重试，服务端不得重复入库 |
| 用户离线期间被移出群组 | 补发消息失败并提示权限已失效 |
| 文件直传失败 | 自动降级为节点中转或对象存储 |
| 分片上传中断 | 保留已完成分片，恢复后继续 |
| 文件哈希一致但用户无权限 | 不允许访问，只复用物理存储 |
| Redis 不可用 | 在线状态等功能降级，核心持久化消息按设计保护 |
| 文件预览服务不可用 | 提供下载选项 |
| 多设备状态冲突 | 服务端状态优先，保留必要的本地副本 |
| 同步数据冲突 | 自动规则解决或进入人工处理列表 |
| 磁盘空间不足 | 停止大文件上传并告警，不影响已有文本消息查询 |
| Token 过期 | 尝试安全刷新，失败则重新登录 |
| 设备被强制下线 | 立即断开连接并清理访问凭据 |
| 临时房间到期 | 按策略冻结、归档或销毁 |

## 13. MVP 验收标准

MVP 至少满足以下条件：

1. 两名用户可在同一局域网连接同一节点并完成私聊；
2. 群组成员可实时收发文本消息；
3. 消息发送后能够收到服务端 ACK；
4. 客户端重复发送同一消息不会产生重复记录；
5. 客户端断开后发送的文本消息进入离线发件箱；
6. 恢复连接后能够补拉遗漏消息并补发待发送消息；
7. 消息顺序以会话序列号为准；
8. 用户被移出群组后无法继续发送群消息；
9. 文件上传和下载需要会话权限；
10. 管理员能够查看在线设备和强制下线；
11. 节点断开公网后，局域网用户仍能完成核心聊天；
12. 系统可以通过 Docker Compose 启动必要依赖；
13. 核心业务具备自动化测试；
14. README 明确区分已实现、部分实现和规划功能；
15. 性能数据必须来自真实压力测试，不使用未经验证的指标。
16. 分片重复提交不重复计数，查询接口能返回已完成分片并继续缺失分片；
17. 完成上传前必须校验缺片、总大小、完整 SHA-256 和文件内容，失败时不得创建文件授权；
18. LOCAL 与 MinIO 可按配置切换，历史 LOCAL 文件不因切换默认存储而失效；
19. 两个共享 MySQL、Redis、MinIO 的应用实例之间可以路由实时事件并聚合 Presence；
20. Redis 短暂失败后，MySQL 中已提交的可靠消息可在重连时通过 `SYNC_REQUEST` 补齐；
21. 文档不得把上述多实例能力描述为独立数据库节点双向复制或冲突合并。

## 16. 风险与约束

### 16.1 浏览器能力限制

纯 Web 客户端难以完成完整的局域网自动发现、后台常驻和系统级截屏检测。相关能力应在桌面客户端或移动客户端中实现，Web 端采用节点地址、二维码和浏览器通知作为替代。

### 16.2 WebRTC 连接限制

设备直传可能受到浏览器兼容性、网络隔离、防火墙和 NAT 策略影响，因此必须设计节点中转降级路径，不能把直传作为唯一文件传输方式。

### 16.3 数据同步复杂度

独立数据库节点之间的双向同步会引入冲突、权限传播、重复数据和安全边界问题。V2.3.0 只实现共享 MySQL、Redis、MinIO 的单逻辑节点多应用实例，不进行节点间数据复制；后续必须先设计同步任务、幂等、权限优先级和冲突处理，再逐步增加手动或自动同步。

### 16.4 安全风险

私有部署不等于天然安全。弱密码、默认密钥、错误权限、日志泄漏和开放文件目录仍可能造成数据泄漏，必须通过配置校验、权限测试和审计降低风险。

### 16.5 项目规模风险

不建议在可靠消息、离线恢复和文件权限尚未稳定前拆分微服务。首期采用模块化单体架构，待压力测试和部署需求明确后再决定是否拆分。
