# LanChat V2.3.0 实施状态

更新日期：2026-07-16

本文把《需求分析-LAN-first-V2.0》和《功能分析-LAN-first-V2.0》的目标能力映射到当前 V2.3.0 代码。状态只描述已有证据，不把设计稿、性能目标或独立节点同步方案算作完成结果。

## 1. 本轮交付范围

当前完成的是 P0 可靠消息纵向切片、V2.1 工程化增强、V2.2 三项协作能力，以及 V2.3 存储和多实例增强：

1. 统一私聊与群聊会话标识；
2. `clientMsgId` 幂等、会话序列号和事务提交后 ACK；
3. 按序列游标查询历史与重连增量补拉；
4. IndexedDB 本地消息缓存与文本离线发件箱；
5. WebSocket 连接后认证、心跳、指数退避和设备会话吊销；
6. Refresh Token 轮换与 HttpOnly Cookie；
7. 文件内容识别、会话权限、物理对象授权、短期签名预览与访问审计；
8. 强配置校验、首次管理员引导和容器隔离的私有部署；
9. 用户连接路径与管理员依赖/JVM/WebSocket 诊断；
10. 服务端 mDNS 广播/发现、节点握手与登录页节点切换；
11. 控制台/滚动文件双日志、管理员筛选查看、错误说明和活动日志导出；
12. WebRTC DataChannel 优先、WebSocket 信令和服务端中转降级；
13. 临时房间码、成员策略、有效期和冻结/归档/销毁生命周期；
14. 广播发布授权、好友复选、在线推送/离线补拉、回执统计和撤销；
15. 上传会话、分片幂等、缺片查询、断点续传、完整哈希与内容复核；
16. LOCAL/MinIO 私有文件对象存储切换和历史 LOCAL 对象兼容；
17. 共享 MySQL、Redis、MinIO 的多 Spring Boot 实例实时路由与全局 Presence；
18. 后端测试、前端类型检查/生产构建和 Compose 配置校验。

## 3. 数据模型变更

### 新增

- `conversation`：统一私聊、群聊以及未来临时房间/广播的顶层模型；
- `conversation_member`：角色、最后已读序列、未读数和离开时间；
- `file_access_grant`：把物理文件存在与用户引用权限分开。
- `file_access_log`：记录文件预览、下载、内容读取和拒绝结果，不保存令牌或正文。
- `file_transfer`：记录直传/中转路由、双方设备、哈希、过期时间和完成状态。
- `temporary_room` 与复用的 `conversation_member`：前者保存临时房间策略，后者保存房间成员角色和生命周期成员状态；没有独立的 `temporary_room_member` 表。
- `broadcast`、`broadcast_receiver`：广播正文、固定接收快照、送达/查看/确认状态。
- `user.can_send_broadcast`：普通账号的广播发布权限；管理员始终允许。
- `file_upload_session`：记录客户端幂等键、会话权限、文件声明、分片布局、存储类型、状态和过期时间。
- `file_upload_part`：记录上传会话内已完成分片的编号、大小、哈希和私有暂存对象。
- `file_object_cleanup_task`：在业务事务中持久化无主文件/分片对象的删除意图，提交后尝试清理，失败后按退避时间重试。
- `file_metadata.storage_type`、`file_metadata.file_path`：前者记录 LOCAL/MinIO，后者保存本地相对路径或 MinIO 对象键；历史记录按 LOCAL 读取。

### `chat_message` 新字段

- `client_msg_id`
- `conversation_id`
- `sequence`
- `sender_device_id`
- `client_created_at`

已有数据库必须依次执行 V2.0、V2.1，V2.2 的文件传输、临时房间、应急广播和广播权限迁移，以及 `sql/migration-v2.3-resumable-object-storage.sql`；广播权限迁移必须位于应急广播迁移之后。V2.3 迁移会把既有文件元数据回填为 LOCAL，并新增上传会话、分片和持久化对象清理任务，不自动搬运已有文件。跨实例实时路由只使用 Redis，不新增数据库表。迁移前需备份；新环境直接使用 `sql/init.sql`。

## 4. 关键工程取舍

### 单逻辑节点、多应用实例

当前仍保持模块化单体，不拆分微服务。V2.3 允许一个逻辑 LanChat 节点运行多个 Spring Boot 实例：业务数据共享 MySQL，文件共享 MinIO，实时事件和 Presence 使用 Redis。WebSocket Session 只存在于接入它的 JVM，其他实例通过 Redis Pub/Sub 把事件送到该 JVM。

Redis Pub/Sub 是实时快路径而不是持久日志。Redis 短暂失败时，本实例连接继续工作，跨实例瞬时事件会降级；可靠聊天消息以 MySQL 会话序列为最终事实，并在重连后通过 `SYNC_REQUEST` 补齐。当前不能宣称支持两个独立数据库节点的双向复制、冲突合并或容灾切换。

### 文本先离线

文本消息可以在无网络时进入 IndexedDB 发件箱。文件仍需连接节点创建/查询上传会话，但上传中断后可以通过 `uploadId` 查询已完成分片并跳过它们继续上传；这不等于完全离线保存文件任务。

会话目录和最近消息也保存在 IndexedDB，因此已认证标签页断网刷新后仍可进入本地会话。完全关闭浏览器会清除 `sessionStorage` 中的短期 Access Token；当前版本没有本地 PIN/设备密钥解锁机制，此时需要重新连到节点完成认证后才能进入缓存，不能宣称“冷启动完全离线登录”。

### ACK 的含义

`CHAT_ACK` 表示服务端已经完成权限校验和数据库事务提交，不代表对方已经阅读。发送方重试相同 `clientMsgId` 时返回原消息信息，不重复落库。

### 文件秒传权限

`/file/check` 只有在用户已经拥有该物理文件的访问权时才返回秒传结果。用户上传完整文件并由服务端重新计算 SHA-256、核对扩展名/MIME/文件头后，可以获得 `UPLOAD_PROOF` 授权。仅知道哈希或存储名不能引用文件。网页与可执行扩展名默认拒绝，签名地址访问时重新复核权限并写审计记录。

### 分片与对象存储边界

分片接口只接收原始二进制和该分片的 SHA-256。重复提交同一编号且内容一致时保持幂等，大小或哈希冲突时拒绝。完成上传时重新检查缺片、完整大小、完整 SHA-256 和文件内容；只有完成事务成功后才产生可引用文件。存储模式属于每个文件元数据，切换默认模式不会把既有 LOCAL 文件误读为 MinIO 对象。临时分片和失去元数据引用的文件对象通过事务内写入的持久化清理任务删除，存储故障不会让删除意图随日志一起丢失。

### 浏览器发现边界

Java 服务端已经能够通过 mDNS 广播和发现同类节点，登录页会展示服务端发现结果。浏览器不能直接监听 UDP 5353，因此首次访问仍需要一个已知节点地址；Docker bridge 网络下也需要 host 网络或多播转发，不能把服务端能力误述为浏览器零配置冷启动发现。

## 5. 当前可验证验收项

- 相同发送者重复提交同一 `clientMsgId` 不会生成第二条消息；
- 每个会话的消息以唯一递增 `sequence` 排序；
- WebSocket URL 不含访问令牌，10 秒内未发 `AUTH` 会断开；
- 设备被下线后，后续事件或心跳会收到强制退出；
- 浏览器断网时可继续发送文本并看到待发数量；
- 恢复连接后先补拉缺失消息，再按创建时间补发本地队列；
- 单次同步不足以返回全部消息时会继续按新位置分页；
- Refresh Token 不出现在 JSON 和 Web Storage；
- 普通令牌过期会保留属主隔离的离线发件箱，重新认证同一账号后继续；明确退出或设备强制下线会清理本地数据；
- 无会话权限不能上传附件，无文件权限不能发送附件消息；
- 扩展名、声明 MIME 与文件头不一致时上传失败，权限撤销后旧签名地址不可继续访问；
- 私有模式使用开发密钥、弱依赖密码或弱管理员密码时启动失败；首次启动只创建一次 `admin`，后续重启不覆盖密码；
- 用户可查看路径、心跳、同步和发件箱诊断，管理员可查看依赖、存储、JVM 与 WebSocket 指标；
- mDNS 解析测试只接受有效节点 ID，并把发现结果转换为校验后的 HTTP(S) 应用地址；
- 在线列表不会返回密码和设备令牌；
- WebRTC 协商失败或超时后会切换服务端中转，未完成任务会按过期时间清理；
- 临时房间过期后按策略冻结、归档或销毁，客户端同步进入只读/不可用状态；
- 未获授权的普通账号不能发布广播，指定接收者必须是发布者自己的有效好友；
- 管理员不进入广播目标人数、不提交回执，可查看统计并撤销广播且保留历史。
- 上传会话只能由创建者在原会话权限仍有效时查询、上传分片、完成或取消；相同分片重试不重复计数；缺片、总大小、最终哈希或内容识别失败时不能生成文件授权；
- LOCAL 与 MinIO 使用相同的私有对象接口，MinIO Bucket 不对浏览器公开；切换默认存储后历史 LOCAL 文件仍可读取；
- 同一逻辑节点的用户分布在不同应用实例时，聊天、业务通知、设备下线和 WebRTC 信令可以跨实例投递；
- 全局 Presence 使用 Redis TTL 和周期续约，实例异常退出后状态自动过期，不在 Redis 中保存 Token 或完整用户正文；
- Redis 短暂失败不改变 MySQL 中已提交消息，客户端重连后按 `sequence` 补拉；独立数据库节点之间不会自动复制或合并冲突。

## 6. 下一阶段建议

按价值和依赖关系推进：

1. Testcontainers + 两个应用实例/两个 WebSocket 客户端的真实 MySQL、Redis、MinIO 集成测试；
2. Redis Streams 或 Outbox 持久事件、跨实例 Trace 与积压监控；
3. 独立节点同步任务、权限传播、冲突检测与冲突合并；
4. WebRTC 传输速率统计和独立节点间信令策略；
5. 团队目录及广播接收候选扩展；
6. 杀毒/内容审核、Actuator、Prometheus、Grafana 和实际压测数据。

在这些能力完成并经过测试前，不应在简历或 README 中声称“5000 条/秒”“99.9% 可用”“独立节点双向同步”“跨独立节点 WebRTC”或“生产级”。可以准确描述为“同一逻辑节点内的多应用实例实时路由”。
